Making a Licq Release

This page means to document all the steps needed for building an offical Licq package and releasing it.

Prerequisites

The following is needed to create a package

  • GIT client and build environment for normal Licq development.
  • GnuPG installed and a default key created (this will be used to sign the package).

The following is needed to create an offical release:

  • GIT access with privileges to commit changes.
  • Access to Licq project on SourceForge with privileges to upload new packages.
  • Membership in Licq mailing lists to post announcement in licq-announce, licq-dev and licq-users.

Preparations

The following things should be ready before releasing a new version:

  • Move the old README file to doc/README-<oldversion> and create a new README file.
  • Update the doc/CHANGELOG file.
  • Update the Licq version in the build files. Note that this must be done for the daemon and for each plugin.
    For Licq this is done in the file licq/cmake/LicqVersion.cmake and for plugins in the file <plugin>/version.cmake (using scripts/set-version makes it easier).
  • Update the Licq version in licq/doc/README and licqweb/README.
  • Make sure the copyright year in licq/LICENSE includes the current year.

Building a Package

To create a package, perform the following steps:

  1. Clone the Licq repository if you don't already have a source tree.
  2. Create a git tag.
  3. Create tarball from the git tag and sign it.
  4. Push the tag.

Full Example:

# git tag -a licq-1.5.1-rc1 1.5-stable
# ./scripts/create-licq-tarball -b -s -t licq-1.5.1-rc1
# git push --tags

Distributing and Announcing

An official release should be distributed and announced at the following locations:

  • Upload packages and sign files to SourceForge or have someone else do if it you do not have access yourself.
  • Update the download page with links to the new files.
  • Update the front page with the latest version.
  • Create a news entry for the start page announcing the new release.
  • Post an announcement to the Licq mailing lists licq-announce, licq-dev and licq-users.
  • Create news entry on SourceForge.
  • Announce the new version on FreshMeat.

Release Plan

First and most importantly, a release should only be performed when the release branch is stable and fully usable. Wait for ongoing changes to be completed rather then releasing in the middle of a high activity period.

Make a first release candidate (package with suffix -rc1) and release according to instructions above.

When a release candidate has been released a code freeze should be in place with the following exceptions:

  • Bug fixes (Important corrections must be performed before the final release.)
  • Translations (Translation files can be updated without affecting the release and translators may need the extra time as well.)
  • Editorial changes (Comments and documentation files can be modified without affecting the release, however avoid changing translatable text strings as this might affect translations.)

After two weeks has passed, either make a new release candidate (-rc2) or a final release.
A guideline for which to do is that if no bugfixes (or other code changes) has been made since the last release candidate then a final release can be made.
If any changes were needed, make a new release candidate instead. If for some reason the first release candidate had serious problems you might want to make the second release candidate sooner than the planned two weeks.

If release candidate two was needed, wait another week and check again if a final release can be made or if a third release candidate is needed.
It should be enough to wait two weeks only after the first release candidate. After the second release candidate wait one week between each release.
